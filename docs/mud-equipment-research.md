# MUD 装备系统调研笔记

这份笔记总结对经典 MUD、现代 MUD、中文/武侠 MUD、MUD 掉落表和文本 UI 装备可读性的调研。它和 `loot-itemization-research.md` 的侧重点不同：那篇看 ARPG 的受控随机，这篇专门回答 **MUD 游戏的装备应该怎么做**。

核心结论：**MUD 不适合 Diablo-like 的大量随机词缀筛选。MUD 的操作和阅读成本天然很高，装备属性应该短、清晰、可解释，核心差异要足够强。掉落表、区域来源、怪物身份、任务奖励和宝箱内容值得认真设计。**

## 总体结论

MUD 装备系统应该更接近：

```text
手工设计的来源掉落表
+ 简洁的装备模板
+ 少量高区分度核心属性
+ 明确的装备槽位和用途
+ 可选的鉴定/评估/比较层
+ 后期再加入合成、贴附、强化、重铸
```

而不是：

```text
海量随机词缀
反复对比 0.7% 提升
每个怪都掉一堆低价值装备
玩家不断 inspect/compare 垃圾
```

适合我们的方向是：

> 装备首先是世界内容，其次才是数值载体。

一件装备最好能用一句话讲清楚：

```text
黑松猎弓：慢速高伤，对野兽有效，但怕雨。
```

而不是：

```text
蓝色弓，评分 43.7，+2.1% 命中，+3.4% 野兽伤害，-1.2% 耐久消耗。
```

## 经典 MUD 的装备结构

经典 Diku/Circle/ROM/SMAUG 风格 MUD 通常使用对象原型和实例字段来描述装备。

常见字段包括：

```text
vnum / id
aliases
short description
long description
object type
extra flags
wear flags
weight
cost
timer / decay
condition
material
level
values by type
affects / modifiers
```

不同物品类型的 `value[]` 含义不同。例如武器、护甲、容器、药水、卷轴各自解释这些数值。

### Wear flags 和 Wear positions

经典 MUD 通常把“能穿在哪里”和“实际穿在哪里”分开。

常见槽位：

```text
finger
neck
body
head
legs
feet
hands
arms
shield
about
waist
wrist
wield
hold
```

后来的 MUD 会扩展出：

```text
back
face
eyes
ears
ankle
dual_wield
missile
tool
```

可借鉴：装备槽位应该清楚，但不要一开始过多。槽位本身也应是结构化能力，不要把所有规则写死在命令里。

### Affects / Modifiers

经典 MUD 常用 affects 修改角色属性，例如：

```text
+strength
+hitroll
+damroll
+armor
+saving throw
+dexterity
```

这套系统简单，但容易产生“堆数值”问题。我们的项目不应该照搬大量细碎 affect，而应该让装备有更少、更鲜明的机械含义。

### Identify 和 Compare

经典 MUD 往往通过 `identify` 显示详细属性，通过 `compare` 给出“比当前装备更好/更差/差不多”的结论，而不是要求玩家手工比较所有数字。

这点非常重要：MUD 的 UI 成本高，系统应该帮玩家做初步判断。

推荐：

```text
look:
  看描述和少量明显特征

appraise / identify:
  看详细属性、条件、耐久、效果

compare:
  和当前同槽位装备比较，输出自然语言结论
```

## 经典 MUD 的掉落方式

经典 MUD 的掉落并不主要来自“死亡时全局随机表”。它更多来自区域文件和 reset 系统。

常见 reset 命令：

```text
M: 加载 mobile
O: 把 object 放入房间
G: 给 mobile 一个 object
E: 给 mobile 装备 object
P: 把 object 放入容器
D: 设置门状态
R: 移除房间物品
```

这意味着很多装备的来源是手工设计的：

```text
某个怪物装备着某把剑
某个箱子里放着某件物品
某个房间刷新某个宝物
```

怪物死后，身上的装备和背包物品进入尸体，玩家再从尸体中拿取。这种设计比“所有怪物共用随机装备池”更有世界感。

可借鉴：

```text
Loadout:
  怪物生成时携带/装备什么

Corpse Loot:
  怪物死后实际掉下什么

Drop Table:
  死亡、搜索、打开箱子、完成任务时额外生成什么
```

这三者应该分开。

## 现代 MUD 的装备经验

### Evennia / EvAdventure

Evennia 的示例游戏强调装备 handler、槽位、背包容量、物品统计展示。它的价值不在于具体规则，而在于分层：角色持有装备状态，装备槽位处理冲突，显示函数负责输出关键数据。

可借鉴：

- 装备槽位由 handler 管理。
- 物品显示应该有统一函数。
- 物品质量可以表示 condition，而不一定是 rarity。

### Aardwolf

Aardwolf 有较强的装备比较、识别、标签化输出和用户偏好。它的 `compare` 思路很值得借鉴：比较不应该只是硬编码评分，而可以允许玩家调整权重。

可借鉴：

- `compare` 可以回答“这件是否适合我”。
- 高级玩家可以配置比较偏好。
- 机器可读/结构化输出可以服务客户端，但默认文本要简单。

### Alter Aeon

Alter Aeon 的装备需求和来源较丰富，包括等级、职业、阵营、诅咒、boss、箱子、任务、重置等。它强调装备来自世界中的具体内容。

可借鉴：

- 装备可以有明确使用要求。
- Boss、箱子、任务和区域重置都是重要来源。
- 鉴定可以短输出，不必每次长篇大论。

### Discworld MUD

Discworld 的 `appraise`、`judge`、`vurdere` 等思路很适合文本游戏。它把“看物理属性”和“判断战斗好坏”区分开，并且评估准确度受技能影响。

可借鉴：

- `look` 看描述。
- `appraise` 看材质、用途、尺寸、状态。
- `judge`/`vurdere` 根据技能判断武器/防具好坏。
- 信息可以是定性的，不一定直接显示数值。

### Achaea / IRE 类游戏

商业 MUD / MUSH-like 游戏通常把 artefact 作为高价值、近乎永久、强身份装备。它们往往不是普通随机掉落，而是购买、任务、活动或特殊系统产物。

可借鉴：

- 顶级装备应该有清晰身份和来源。
- 稀有装备不一定是随机词缀更多，而是功能更明确、生命周期更稳定。

### LuminariMUD / CoffeeMUD

这类系统有更现代的随机 treasure、material、rarity、creation points、generated description 等。它们说明 MUD 可以支持随机生成，但仍应通过 base type、material、rarity、预算和描述系统控制。

可借鉴：

- 随机装备应有生成预算。
- 材质和基础类型很重要。
- 自动生成描述比裸露属性更适合文本游戏。

## 中文 / 武侠 MUD 的装备经验

公开中文 MUD 资料和开源 mudlib 显示，传统中文/武侠 MUD 常见装备结构相对直接。

### 武器和防具

常见武器定义包含：

```text
name
id
weight
unit
description
value
material
wield message
unwield message
```

防具常见：

```text
armor_prop
wear message
remove message
material
weight
armor value
dodge penalty
```

这种设计很朴素，但对文本 MUD 来说有优势：读起来直接。

### 装备耐久

中文 MUD 中耐久/损坏并不少见。例如装备可能因使用而降低耐久，过低时自动卸下或损坏。

可借鉴：

- 耐久是限制强装备的重要方式。
- 但强装备不应轻易永久消失，除非这是明确设计。
- 对长期投入装备，应有修理、保护、绑定或降级机制。

### 玩家制造和签名

一些中文 MUD 支持玩家打造装备，并记录制造者、描述、材料、属性点数等。这很符合中文 MUD 的社交和长期养成特征。

可借鉴：

- 物品实例应保存 provenance：谁制造、何处获得、由何材料构成。
- 玩家可以影响名称、描述、铭刻或外观，但需要防止误导。
- 制作装备可以成为后期深度系统，而不是第一版核心。

### 现代中文 MUD 的品质系统

有些现代中文 MUD 会使用类似：

```text
劣质 / 普通 / 精致 / 卓越 / 无暇 / 完美
普通 / 优良 / 珍稀 / 史诗 / 传说 / 神器
粗糙 / 普通 / 精致 / 完美 / 传说
```

同时有分解、升级、精炼、重铸、签名、镶嵌、套装等系统。

这说明中文玩家可以接受装备品质和养成系统，但 MUD 的显示仍然应该避免属性过载。

## 文本 UI 可读性原则

MUD 装备设计不能脱离文本 UI。玩家在命令行里比较装备，比在 ARPG 图形界面里更累。

### 三层显示

推荐所有装备支持三层显示：

```text
列表层:
  一行短名称，少量标记

观察层:
  描述、用途、明显特征、槽位、状态

鉴定/评估层:
  关键属性、限制、效果、耐久、来源、隐藏信息
```

例子：

```text
inventory:
  一柄黑松猎弓

look bow:
  这是一柄用黑松木制成的长弓，弓身偏重，弦声沉闷。
  它似乎特别适合猎杀野兽，但木材怕潮。

appraise bow:
  类型: 双手弓
  特性: 慢速、高伤、猎兽、怕雨
  状态: 良好
  需求: 力量略高
```

### 定性优先，数值后置

默认不应展示一大串数字。更适合文本游戏的是：

```text
很轻
偏重
锋利
迟钝
很耐用
快要损坏
适合刺击
不适合破甲
```

数值可以通过技能、工具、鉴定或专家模式显示。

### Compare 应该替玩家减负

`compare` 不应该把两件装备的所有字段并排打印，而应该回答：

```text
它比你现在用的剑更锋利，但更重，挥动会更慢。
如果你依赖轻攻击，它不太适合；如果你需要破甲，它更好。
```

高级玩家可以选择详细模式：

```text
compare sword detail
```

### Rarity 不是价值

文本游戏尤其需要避免“颜色越高越好”的懒惰设计。稀有度应该表达获得难度、来源和经济含义，而不是直接等于强度。

一件普通铁剑在正确场景中可能比传说仪式匕首更适合战斗。

## 推荐的装备等级/稀有度分层

我们不建议采用 Diablo-like 的：

```text
普通 / 魔法 / 稀有 / 传奇
```

因为这会引导玩家期待大量随机属性比较。

更适合 MUD 的是拆成三个维度。

### 1. 制作品质 / 状态质量

表示做工、保存状态、耐久，不直接等同稀有：

```text
破损
粗制
普通
精良
名作
```

这影响基础性能、耐久、可靠性。

### 2. 稀有度 / 可获得性

表示世界中出现频率和经济稀缺：

```text
常见
少见
稀有
罕见
传说
```

这影响掉落表、流通和收集价值，但不应自动表示更适合当前角色。

### 3. 身份层级

表示它是否有特殊来源和名字：

```text
制式装备
地区名物
命名装备
唯一物品
遗物
```

例如：

```text
制式装备: 城卫长剑
地区名物: 黑松猎弓
命名装备: 灰塔守门人的钥斧
唯一物品: 时空扭曲法杖
遗物: 初火余烬
```

这比单一颜色品质更适合 MUD。

## 推荐的装备属性规模

为了避免阅读负担，每件普通装备应只暴露少量核心信息。

### 武器

建议核心信息：

```text
类型 / 使用方式
伤害倾向
速度或节奏
重量或耐力消耗
1-2 个特性
1 个限制或代价
```

例子：

```text
黑松猎弓
  双手弓
  慢速，高伤
  对野兽有效
  怕雨
```

```text
锈裂短剑
  单手剑
  快速，低伤
  容易造成流血
  对护甲效果差
```

### 防具

建议核心信息：

```text
覆盖部位
防护倾向
重量 / 噪音 / 灵活度
1 个特性
1 个限制或代价
```

例子：

```text
鱼油布衣
  身体防具
  轻，防护低
  滑溜溜
  高度易燃
```

### 饰品

饰品最适合承载一个明确效果：

```text
一个主效果
一个代价或限制
```

例子：

```text
静默戒指
  让你更难被听见
  也让你施法更困难
```

## 掉落表设计建议

掉落表应该按来源设计，而不是全局随机。

推荐来源类型：

```text
MobLoadout
  怪物生成时装备/携带的东西

CorpseTransfer
  怪物死后身上真实物品进入尸体

DeathDropTable
  死亡时额外生成的掉落

RoomReset
  房间自然刷新物

ChestTable
  宝箱内容

SearchTable
  搜索房间、尸体、废墟时出现的东西

QuestReward
  任务奖励

ShopStock
  商店库存

AreaEventReward
  区域事件奖励
```

每个掉落表可以包含：

```text
fixed entries
common pool
rare pool
empty chance
roll count
max_world / max_area / max_source
cooldown
announce/log policy
team distribution policy
```

### 掉落来源应该塑造物品

例如：

```text
火龙:
  火焰、龙族、高温、鳞片、威压

下水道:
  腥味、毒、潮湿、滑溜溜、腐败

黑松林:
  木质、猎兽、怕雨、树脂、隐蔽
```

这让装备更像世界的一部分，而不是抽卡机。

## 推荐的数据概念

这里讨论的是框架需要支撑的数据边界，不是在现在决定具体装备内容。未来装备系统至少应该能区分：

```text
ItemTemplate
  物品模板，定义基础身份和允许空间

ItemInstance
  具体实例，保存 roll、耐久、历史、来源

DropTable
  掉落表

DropSource
  掉落来源，例如怪物、宝箱、区域、任务

Loadout
  怪物生成时携带或装备的物品

Provenance
  物品来源记录：谁掉落、谁制造、哪里获得、何时生成
```

即使第一版不实现完整随机，也不要把物品写成只有静态定义、没有实例和来源。

### 框架需要准备好的部分

框架层应该提供这些能力：

```text
ItemTemplate / ItemInstance 的分离
稳定的实例 ID
装备槽位或装备位置能力
物品来源记录的基本位置
DropSource / DropTable 的数据结构
Loadout、尸体转移、宝箱、任务奖励等来源类型
list/look/appraise/compare 的展示分层
少量通用数值和 tag 的显示接口
内容加载校验
```

这些是支撑装备系统的基础设施。没有它们，后续内容会很难扩展。

### 现在不应该决定的内容设计

以下内容属于未来内容设计，不应该在框架阶段定死：

```text
具体有哪些品质名称
每个区域掉什么
每个怪物的掉落表
每件武器的伤害/速度/特性
稀有度具体概率
顶级装备的名字和效果
某个材料是否应当稀有
某个装备是否应该怕雨、猎兽、诅咒或赐福
```

框架只需要允许这些内容被清楚表达、加载、校验、持久化和展示。具体内容应该等世界观、区域、战斗模型和玩家成长节奏更清楚后再设计。

## Builder / 内容作者校验

为了让框架支撑“短而有区分度”的装备，内容工具或加载器应该提供校验能力。具体阈值可以以后再定，但校验方向包括：

```text
属性数量是否过多
是否缺少来源
是否没有装备槽位
稀有度和掉率是否矛盾
唯一物品是否缺少流通策略
装备是否只有数值没有身份
描述是否暗示了实际不存在的能力
tag 是否误导玩家
```

这和我们之前的原则一致：tag 是玩家可学习、可信任的语义单位。

## 与 Diablo-like 的区别

我们不是完全否定 ARPG 装备系统。前一篇调研已经说明，ARPG 也在做受控随机。但 MUD 需要更进一步减少阅读负担。

区别如下：

```text
Diablo-like:
  高频掉落
  大量属性比较
  UI 支持快速筛选
  玩家接受刷装备循环

MUD-like:
  文本操作繁琐
  装备比较成本高
  来源和描述更重要
  掉落应少而有意义
  核心属性要短而鲜明
```

因此我们的方向应是：

```text
少掉落
强来源感
短属性
高区分度
系统帮助 compare
后期通过合成/贴附/强化增加深度
```

## 对第一版的影响

第一版不需要实现完整装备系统，更不需要决定完整装备内容。但可以从一开始遵守这些框架边界：

```text
物品有 template_id 和 instance_id
物品实例有保存来源记录的位置
装备槽位是能力，不是硬编码分类
掉落从具体来源表产生，而不是全局随机池
装备显示分 list/look/appraise 三层
compare 输出自然语言建议
不要生成大量随机小属性
```

第一版最小装备系统可以只有：

```text
武器
防具
手持物
少量槽位
耐久
1-2 个核心 tag
手工掉落表
```

等核心 MUD loop、持久化和来源掉落跑通后，再考虑品质、强化、合成、贴附和随机生成。

## 当前建议

目前应该确定的是框架方向，而不是具体装备内容：

```text
来源设计优先
手工掉落表优先
短属性优先
核心差异优先
稀有度不等于强度
品质、稀有度、身份分层
比较负担由系统承担
随机生成作为后期扩展
```

这些不是在规定第一批装备必须长什么样，而是在规定框架不能只支持 Diablo-like 的大量词缀随机。内容设计可以以后慢慢做，但框架现在就应该避免把未来锁死。

最终目标不是让玩家在垃圾堆里找 1% 提升，而是让玩家记住：

```text
这把弓来自黑松林，射得慢但很痛，猎兽很好，雨天很糟。
```

这种装备更适合 MUD。
